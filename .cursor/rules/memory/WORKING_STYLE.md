# Working Style Directions

Last updated: 2026-04-20 (session 4: P1.8 of display library refactor — added immutable-defaults directive, decompose-into-small-helpers directive, and `working-docs/` folder directive; four of P1.8's seven requested items were already captured at a more abstract level in Core Principles and were not duplicated, per abstraction-lifecycle rules)

**Abstraction convention**: entries start specific when a concept is new and evidence is narrow ("in situation X, Y worked"). As multiple similar-but-not-identical situations accumulate, related entries are consolidated into a generalized principle with the specific incidents preserved as evidence in the Notes column. Consolidation is a deliberate compaction, never a side effect.

**Scope tags** (used in the Scope column of every directive table):

| Tag | Meaning |
|-----|---------|
| `[universal]` | Applies to any AI–human collaboration, any project, any domain |
| `[user]` | Specific to this user's preferences/conventions; carries across their projects |
| `[project]` | Specific to this project (Exp14 display library, CircuitPython on YD-RP2040) |
| `[task]` | Specific to a current task or phase; rotates frequently |

When importing a directive observed in a different setting, re-evaluate its scope against the current context — assumptions about authority, code maturity, and risk profile do not transfer automatically (see Core Principle: scope-transfer check).

**Status notation** (only when not the default `established`): inline prefix `(experimental)` in the Direction column for newly-derived self-prescribed directives that have not yet been tested in unseen-but-similar situations. Promotes to `established` once it has been applied successfully without correction across at least two distinct situations beyond the original derivation. The Reinforcements counter tracks raw incident count; Status tracks confidence at the entry's current abstraction level.

## Code Editing

| Direction | Scope | Reinforcements | Last Applied | Notes |
|-----------|-------|----------------|--------------|-------|
| **Immutable defaults go directly in the signature.** Parameters with immutable default values (`int`, `str`, `tuple`, `bytes`, `frozenset`) take the literal as the default — e.g. `color=WHITE`. The `None`-sentinel-plus-remap pattern (`color=None` followed by `if color is None: color = WHITE` inside the body) is reserved for *mutable* defaults (`list`, `dict`, `set`, `bytearray`) where the shared-instance hazard is real. Applying `None`-remap to immutable values is cargo-culted indirection — it adds a line of dead code and obscures the actual default. **Evaluate**: "Is the default value mutable?" **Act**: if yes, use `None`-remap; if no, put the default directly in the signature. | `[universal]` | 1 | 2026-04-20 | Triggered during `display_library_refactor_*.plan.md` P1.2: 13 sites in `lib/display/core.py` used `color=None` + `if color is None: color = WHITE`. All replaced by direct `color=WHITE` with no behavior change (Tier 1 tests green). Retrospective L1 will check whether any such site in this project actually had mutable-default intent that we missed. |
| **(experimental) Decompose procedural logic into small, self-contained helpers.** Target ~≤ 20 executable lines per helper. Each one has: (a) a single responsibility expressible in one sentence, (b) a named invariant callers can rely on, documented inline, (c) minimal internal checks — just enough to catch catastrophic drift (e.g. `assert len(ring) == 3 * WIDTH`), no defensive revalidation of inputs already validated at module boundaries, no per-frame type/shape checks in hot loops, (d) independent testability where Tier 1 permits. **Goal**: make invariants visible and shift the mental model from "trace the whole procedure" to "verify each block's contract locally." Long procedural functions bury cancellation / buffer-boundary / off-by-one invariants. **Evaluate**: "What invariant does this block enforce, and can I name it?" **Act**: if the invariant is buried, extract a helper and write the invariant at its docstring. | `[universal]` | 1 | 2026-04-20 | Articulated as the cross-cutting directive for Phase 2 of `display_library_refactor_*.plan.md`; targets P2.1 (parser dedup), P2.4 (range-split), P2.5 (ring buffer). Status `experimental` at the abstracted form because it has not yet been applied to refactoring work outside this plan; re-evaluate at Phase 2 close with concrete examples and promote to `established` after at least two clean applications beyond this plan. |
| **(experimental) Rewrite-vs-edit trade-off for early-phase library code.** **Goal**: choose the change shape (small edit, scoped refactor, full rewrite of a section) that produces the best outcome on clarity / robustness / maintainability / functionality / usability for the user's review effort — not the smallest diff by reflex. **Default posture**: in early-phase library development, larger rewrites of a *section* are often the right move when they consolidate logic, eliminate near-duplication, or materially improve clarity/robustness/usability. **Constraints on rewrites**: (a) state the goal up front before making the change; (b) keep the rewrite *localized* — touch only the section being improved, leave functionally independent adjacent code alone even if it looks tempting; (c) consolidate or deduplicate only when the bloat or near-duplication is real, not cosmetic; (d) do not change behavior without a clear, named benefit. **Trade-off awareness**: every rewrite costs the user review effort. Re-evaluate frequently — when the marginal clarity/robustness gain is small relative to review cost, prefer an edit. **Promotion path**: as the library stabilizes (post-MVP, public-API stable, on-device behavior verified end-to-end), the constraint side tightens — minimal-invasiveness becomes the default and rewrites need stronger justification. Re-evaluate at phase boundaries. | `[project]` | 1 | 2026-04-20 | Stated by the user as a deliberate counter to a more conservative "edits over rewrites" posture I had proposed importing. ## Retention and Evaluation

- Compaction is always a deliberate activity, never a side effect of another edit.
- Never silently drop directives. Absence of recent corrective feedback means a directive is probably working.
- Before removing any entry: state what, why, and verify it isn't the sole record of tracking metadata.
- Prefer splitting over pruning. Create topic files; don't delete to shrink.
- **Abstraction lifecycle**: entries enter specific ("in situation X, Y worked") and are lifted to generalized principles once multiple similar-but-not-identical situations accumulate and a common underlying mechanism becomes visible. When consolidating, the abstract principle replaces the specific entries in the Principle column, and the specific incidents move to the Notes column as dated evidence. Reinforcements count then equals the number of consolidated incidents. Nothing is lost; only the level of description changes. A freshly-lifted abstraction inherits the incident count but gets `(experimental)` Status — the abstract form has not yet been tested on a *new* situation, only on the situations that produced it; promote to default Status after at least two clean applications on situations outside the originating set.
- **Generalization stopping rule**: when ascending the abstraction ladder (specific incident → task type → domain → universal), stop one level *below* either of these triggers: (a) a counterexample at the next level is articulable, or (b) the principle becomes so abstract that it loses operational guidance — i.e., the Evaluate/Act lines no longer suggest a concrete next move. Over-generalization is the mirror failure of under-generalization: both cost evaluability.
- **Single-incident entries are still legitimate** when they describe a stated preference or a clearly-scoped convention (e.g. commit message format, line length). Not everything needs to lift to an abstract principle — only items where abstraction makes the rule more portable to unseen-but-similar situations.
