## Purpose

Directives about *how code itself should be written* — API shape, code structure, correctness-argument hygiene, test construction, refactoring economics. Adjacent to:

- `WORKING_STYLE.md` — how I *collaborate* (communication posture, process, judgment, artifact conventions). A directive like "commit message format" or "re-read files before edit" lives there.
- `TECHNICAL.md` — what I *know about the domain* (CircuitPython internals, board pinouts, library APIs). A directive like "modulo is expensive on M0+" lives there.

This file sits between the two: directives that transcend a single domain but pertain to the shape of code artifacts rather than to collaboration behavior or domain facts.

**Schema** — inherited from `WORKING_STYLE.md`:

- **Abstraction convention**: entries start specific ("in situation X, Y worked"); lifted to generalized principles when multiple similar-but-not-identical situations accumulate a common underlying mechanism. Specific incidents move to the Notes column as dated evidence. Consolidation is deliberate, never a side effect.
- **Scope tags** (`[universal]` / `[user]` / `[project]` / `[task]`): same meanings as the legend in `WORKING_STYLE.md` header. Re-evaluate scope when importing a directive observed in a different setting.
- **Status notation** (only when not the default `established`): inline `(experimental)` prefix for newly-derived directives that have not yet been tested on unseen-but-similar situations. Promotes to `established` after at least two clean applications outside the originating incident.

| **(experimental) Code expresses high-level structure; comments carry low-level semantics.** **Goal**: keep the structure of the solution visible in the code itself — parallel blocks, consistent naming, shared shapes — so a reader grasps the approach before diving into any single branch. | **(experimental) Test design follows the bug taxonomy, not line coverage.** **Goal**: choose tests that each discriminate a specific class of bug, so the suite as a whole covers the space of plausible failures. Line coverage is a proxy that correlates with but does not equal this — a suite can have 100% line coverage and 0% discriminative power (all tests exercise happy paths; none distinguish correctness from near-correctness). **Bug classes to consider for transform-style code**: (1) coarse — wrong shape, wrong size, wrong type; (2) corner — off-by-one at boundaries; (3) interior — correct at corners but wrong in the middle; (4) permutation-vs-rotation — output has the right set of values but arranged wrong; (5) composition — individually correct operations compose wrongly (e.g. two rotations not equalling their sum). **Shared-derivation hazard**: when the test's reference formula shares a derivation with the implementation, a bug in the derivation infects both and the test can't catch it — break the dependency with a visually hand-constructed reference or an independent algorithm. **Evaluate**: "For each bug class this code could plausibly exhibit, which test in the suite would catch it? If none, is that class worth adding a test for? Does any test's reference share a derivation with the code under test?" **Act**: add tests surgically, one per bug class where feasible. Coarse → corner → interior → relational is a reasonable progression.
