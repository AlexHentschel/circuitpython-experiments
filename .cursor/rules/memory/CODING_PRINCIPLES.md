## Purpose

Directives about *how code itself should be written* — API shape, code structure, correctness-argument hygiene, test construction, refactoring economics. Adjacent to:

- `WORKING_STYLE.md` — how I *collaborate* (communication posture, process, judgment, artifact conventions). A directive like "commit message format" or "re-read files before edit" lives there.
- `TECHNICAL.md` — what I *know about the domain* (CircuitPython internals, board pinouts, library APIs). A directive like "modulo is expensive on M0+" lives there.

This file sits between the two: directives that transcend a single domain but pertain to the shape of code artifacts rather than to collaboration behavior or domain facts.

**Schema** — inherited from `WORKING_STYLE.md`:

- **Abstraction convention**: entries start specific ("in situation X, Y worked"); lifted to generalized principles when multiple similar-but-not-identical situations accumulate a common underlying mechanism. Specific incidents move to the Notes column as dated evidence. Consolidation is deliberate, never a side effect.
- **Scope tags** (`[universal]` / `[user]` / `[project]` / `[task]`): same meanings as the legend in `WORKING_STYLE.md` header. Re-evaluate scope when importing a directive observed in a different setting.
- **Status notation** (only when not the default `established`): inline `(experimental)` prefix for newly-derived directives that have not yet been tested on unseen-but-similar situations. Promotes to `established` after at least two clean applications outside the originating incident.

| **(experimental) Provide a correctness argument alongside every non-trivial transformation.** **Goal**: force the "why is this correct?" question to be answered *before or during* implementation, not deferred to "we have tests, it must be right". Applies to algebraic simplifications, loop-invariant hoisting, refactoring that reshapes control flow, any change that preserves observable behavior while altering structure. **Evaluate**: "Is the step from version A to version B self-evident to a reader who isn't me-right-now? If not, what specific argument would make it evident?" **Act**: capture the argument at the site — inline derivation block, docstring proof sketch, or short comment chain — *as part of* the implementation, not as an afterthought. | **(experimental) Primitive convenience is justified only when the marginal cost to non-users is negligible.** **Goal**: keep each library primitive paying only for work every caller needs. Convenience features (wrap-around on rotation, None-coercion on colors, implicit type promotion, verbose error messages involving string formatting) add cost to every call regardless of whether the caller uses the feature. Acceptable convenience-cost threshold is platform-dependent — what's negligible on desktop Python is not negligible on Cortex-M0+ without a hardware divider. **Evaluate**: "Of this primitive's expected callers, what fraction uses this convenience? For the ones who don't, what's the per-call cost they pay? Can the convenience be replicated as a one-liner at the call site for the users who do want it?" **Act**: if only some callers use it / a one-liner suffices / the per-call cost is non-trivial for non-users, push to the call site and document the escape-hatch idiom in the primitive's docstring. If every caller needs it, keep it and note the cost where relevant. Equality checks and bit-ops are usually below threshold; modulo on M0+, floating-point on no-FPU cores, and heap allocations in hot paths are usually above. | `[universal]` | 1 | 2026-04-21 | Universal form of the MCU-specific directive at `WORKING_STYLE.md § Domain-Specific` ("Library primitives do not bake convenience operations into their default path"). | **(experimental) Strict input validation on cold call sites is free insurance.** **Goal**: catch mis-use at its source when enforcement is effectively free. Cold call sites (module load, `__init__`, configuration, `set_rotation`, mode switches) are invoked at human-scale cadence or rarer; a validation check adds no meaningful runtime cost on the happy path and prevents a silent-wrong-answer failure mode on the unhappy path. **Contrast**: in hot paths (per-frame, per-pixel, per-sample loops), the same check pays every iteration, so the calculus flips toward trust-at-boundary + check-once-on-entry. **Evaluate**: "Is this call site invoked at human / configuration / startup cadence? Is the input domain discrete or small? Does a silent fall-through on invalid input fail safely or mask a bug?" **Act**: on cold sites with discrete input domains, raise on invalid input with a descriptive error naming the accepted set. Do not silently default, do not silently coerce, do not paper over. On hot sites, validate once at a boundary and trust downstream. | `[universal]` | 1 | 2026-04-21 | Derived from `build_lut` refactor: the original implementation silently fell through to `rotation=0` on any unrecognized input. `set_rotation` is called once per orientation change — a cold site by any measure — so the validation cost is zero in practice. | **(experimental) Code expresses high-level structure; comments carry low-level semantics.** **Goal**: keep the structure of the solution visible in the code itself — parallel blocks, consistent naming, shared shapes — so a reader grasps the approach before diving into any single branch. Per-block specifics (what a particular constant's value happens to encode, why one branch's formula has a `+1` the others don't) live in local comments where they can't break structural consistency. Inverse failure mode: naming low-level variables by per-block interpretation (`x_top` in branch A, `x_tail` in branch B, `x_comp` in branch C, `xw` in branch D) fragments what is conceptually the same slot across parallel blocks, forcing the reader to re-derive the structural parallel each time. A corollary: code documentation should not redundantly describe what the code already shows at the structural level; it carries the per-site specifics the code doesn't convey. **Evaluate**: "Do parallel blocks look parallel? Do corresponding slots have corresponding names, even when per-slot semantics differ? Is the high-level structure of the solution readable without reading any single block in detail?" **Act**: prioritize structural consistency in code — same role, same name, across blocks. Push per-block semantics (what the value means, what algebraic simplification yielded it) into local comments at each site. When code-structure consistency and local-name expressiveness conflict, consistency wins and the comment carries the specificity. | `[universal]` | 1 | 2026-04-21 | Derived from `build_lut` refactor: four rotation branches initially had branch-specific names (`x_top`, `x_tail`, `x_comp`, `xw`), each chosen to describe what its value happens to encode in that branch. User flagged that the parallel structure — "each branch computes a per-x offset, then an inner y loop applies it" — was obscured by the naming. | **(experimental) Test design follows the bug taxonomy, not line coverage.** **Goal**: choose tests that each discriminate a specific class of bug, so the suite as a whole covers the space of plausible failures. Line coverage is a proxy that correlates with but does not equal this — a suite can have 100% line coverage and 0% discriminative power (all tests exercise happy paths; none distinguish correctness from near-correctness). **Bug classes to consider for transform-style code**: (1) coarse — wrong shape, wrong size, wrong type; (2) corner — off-by-one at boundaries; (3) interior — correct at corners but wrong in the middle; (4) permutation-vs-rotation — output has the right set of values but arranged wrong; (5) composition — individually correct operations compose wrongly (e.g. two rotations not equalling their sum). **Shared-derivation hazard**: when the test's reference formula shares a derivation with the implementation, a bug in the derivation infects both and the test can't catch it — break the dependency with a visually hand-constructed reference or an independent algorithm. **Evaluate**: "For each bug class this code could plausibly exhibit, which test in the suite would catch it? If none, is that class worth adding a test for? Does any test's reference share a derivation with the code under test?" **Act**: add tests surgically, one per bug class where feasible. Coarse → corner → interior → relational is a reasonable progression. Break shared-derivation dependencies with visual fixtures or independent algorithms. | `[universal]` | 1 | 2026-04-21 | Derived from the `tests/test_geometry.py` final analysis pass: the pre-existing suite had shape, corner-mapping, and `xy_to_index` tests, but no interior check (a bug that preserved corners but flipped interior rows would have passed — Gap 1), no rotation-composition check (a bug that rotated by 90° but didn't compose predictably would have passed — Gap 3). | **Immutable defaults go directly in the signature.** Parameters with immutable default values (`int`, `str`, `tuple`, `bytes`, `frozenset`) take the literal as the default — e.g. `color=WHITE`. The `None`-sentinel-plus-remap pattern (`color=None` followed by `if color is None: color = WHITE` inside the body) is reserved for *mutable* defaults (`list`, `dict`, `set`, `bytearray`) where the shared-instance hazard is real. Applying `None`-remap to immutable values is cargo-culted indirection — it adds a line of dead code and obscures the actual default. **Evaluate**: "Is the default value mutable?" **Act**: if yes, use `None`-remap; if no, put the default directly in the signature. | `[universal]` | 1 | 2026-04-20 | Triggered during `display_library_refactor_*.plan.md` P1.2: 13 sites in `lib/display/core.py` used `color=None` + `if color is None: color = WHITE`. All replaced by direct `color=WHITE` with no behavior change (Tier 1 tests green). Retrospective L1 will check whether any such site in this project actually had mutable-default intent that we missed. |
| **(experimental) Decompose procedural logic into small, self-contained helpers.** Target ~≤ 20 executable lines per helper. Each one has: (a) a single responsibility expressible in one sentence, (b) a named invariant callers can rely on, documented inline, (c) minimal internal checks — just enough to catch catastrophic drift (e.g. `assert len(ring) == 3 * WIDTH`), no defensive revalidation of inputs already validated at module boundaries, no per-frame type/shape checks in hot loops, (d) independent testability where Tier 1 permits. **Goal**: make invariants visible and shift the mental model from "trace the whole procedure" to "verify each block's contract locally." Long procedural functions bury cancellation / buffer-boundary / off-by-one invariants. **Evaluate**: "What invariant does this block enforce, and can I name it?" **Act**: if the invariant is buried, extract a helper and write the invariant at its docstring. | `[universal]` | 1 | 2026-04-20 | Articulated as the cross-cutting directive for Phase 2 of `display_library_refactor_*.plan.md`; targets P2.1 (parser dedup), P2.4 (range-split), P2.5 (ring buffer). ## Retention and Evaluation

Inherits from `WORKING_STYLE.md § Retention and Evaluation`:

- Compaction is always deliberate, never a side effect.
- Never silently drop directives.
- Prefer splitting over pruning.
- Abstraction-lifecycle convention.
- Generalization stopping rule (stop one level below articulable counterexample OR loss of operational guidance).
- Single-incident entries legitimate when describing a stated preference or clearly-scoped convention.

When a cross-applicability pattern emerges between this file and `WORKING_STYLE.md` (e.g. a coding-craft directive that turns out to be a special case of a collaboration principle, or vice versa), consider consolidation via the lifecycle rule — but as deliberate compaction, not a side effect of another edit.
