### Discussion notes

- **#2 font precompute** — open design questions to resolve before committing:
  - Storage shape: flat `bytes`/`bytearray` with a parallel offset table (proportional widths differ per glyph, so not a fixed stride), vs. a `tuple`/`list` of per-glyph `bytes`. Flat + offsets minimizes object count (lower fragmentation per CONTEXT_HANDOFF engineering rules); list-of-bytes is simpler to index.
  - Index scheme: `ord(ch) - 32` into the table; bounds-check + fall back to blank (current `_glyph_columns` returns `bytes(WIDTH)` for missing glyphs).
  - Import-time cost: pairs with retrospective **L2** (import-time allocation strategy) — measure `gc.mem_free()` before/after building the cache, like the Icons-as-Image materialization.
  - Interaction with `_GlyphColumnFeeder`: feeder currently calls `_glyph_columns(ch)` lazily; a precomputed cache would let it index instead. Confirm the feeder's per-char `_current_cols` semantics still hold.
  - Could also memoize lazily (build on first use) instead of eagerly at import — decide eager-vs-lazy on the L2 measurement.
